LZ77: compression by reference

Interactive

History repeats itself.

Huffman and arithmetic coding squeeze a stream symbol-by-symbol. LZ77 attacks a different kind of redundancy: repetition. If a chunk of text appeared earlier, don't spell it out again — just say "go back d characters and copy n of them." It's the engine inside gzip, zstd, zip, and PNG.

Step 1

The idea: point back instead of repeating

LZ77 slides a window along the text. At each position it splits the world into a search buffer (what it has already encoded) and a look-ahead buffer (what's coming). It finds the longest stretch of upcoming text that also appears in the search buffer, and replaces it with a compact token:

(offset, length, next-literal)

— "go back offset characters, copy length of them, then add this one new character." If nothing matches, it emits a token with length 0 and just the literal. Decoding is then trivially fast: you only ever copy.

Matches can overlap themselves! To encode aaaaaaaa, the very first a becomes a literal, then a single token "go back 1, copy 7" reproduces the rest — the copy reads characters it is still writing. That's how LZ77 turns long runs into almost nothing.

Step 3

From tokens to bits

Parsing is only half the job. LZ77 produces three streams — the literals, the match lengths, and the match offsets — and each is then entropy-coded (that's where Huffman, ANS, or arithmetic coding come back in). gzip Huffman-codes them; zstd uses tANS. The size readout above uses a crude fixed-width estimate, so it only beats raw on clearly repetitive text — real LZ77 does far better once the streams are entropy-coded.

Why even small matches pay off: an offset of a million fits in ~20 bits, and a length in a handful more. Replacing an 8-character repeat (64 bits as literals) with a ~30-bit reference is a win even when the match is far away. That's why LZ77 shines on real text and code, which repeat constantly.

Step 4

Why it's a big deal

LZ77 is universal: without being told anything about the source, it provably approaches the best possible compression rate as the input grows — it learns the repetition structure on the fly. Combined with its blazing-fast, copy-only decoding, that's why it (and its descendants LZMA, LZ4, Snappy) sit underneath a huge fraction of the world's stored and transmitted data.
