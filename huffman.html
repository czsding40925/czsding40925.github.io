Huffman Coding

Interactive

The optimal prefix-free code.

We left the intro with a goal: a code that's prefix-free (instantly decodable) and optimal (smallest average length). Huffman coding finds it — with a recipe so simple it fits in three steps.

Step 1

How long should each codeword be?

Entropy already hints at the answer. The "surprise" of a symbol, $\log_2\frac{1}{p_i}$, is exactly the number of bits it deserves: a symbol with probability $\tfrac12$ wants 1 bit, one with probability $\tfrac14$ wants 2 bits, and so on. So the ideal length is

$\ell_i \approx \log_2\dfrac{1}{p_i}.$

Plug those lengths into the average and you get exactly the entropy $H$. The catch: $\log_2\frac1{p_i}$ is usually a fraction, and codeword lengths must be whole numbers.

A first attempt — Shannon coding. Just round up: $\ell_i=\big\lceil\log_2\frac1{p_i}\big\rceil$. This is always prefix-free and provably within one bit of entropy ($\bar\ell < H+1$). But rounding every symbol up independently can waste bits — Shannon coding is good, not always optimal. To truly minimize the average, we need to be smarter about how the lengths fit together.
Step 2

Huffman's recipe

In 1952, a grad student named David Huffman found the optimal answer — and it builds the code bottom-up, starting from the rarest symbols:

1. Start with every symbol as its own little tree, weighted by its frequency.
2. Take the two lowest-weight trees and merge them under a new parent (its weight is the sum).
3. Repeat until a single tree remains.

Reading the path from the root to each leaf — left = 0, right = 1 — gives that symbol's codeword. Because frequent symbols get merged in last, they sit near the top with short codes, while rare symbols sink deep and get long ones. The greedy "merge the two smallest" rule turns out to be exactly what's needed to guarantee the result is optimal.

The more skewed your text (lots of repetition, few distinct symbols), the further Huffman drops below fixed-length — and the closer it hugs the entropy floor. Feed it something uniform like random DNA and all three nearly meet: there's no redundancy to exploit.

Step 4

Huffman's one weakness

Huffman is optimal — but only among codes that give each symbol a whole number of bits. A symbol that "deserves" 1.4 bits still costs 2. Formally, Huffman's average length always lands in $[H,\ H+1)$: within a bit of perfect, but it can't get closer when probabilities aren't powers of two. To break that whole-bit barrier we stop coding symbol-by-symbol entirely — which is the idea behind arithmetic coding.
