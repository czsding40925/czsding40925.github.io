1.1 · Introduction to Lossless Coding

Interactive Notes Original ↗

We use data compression thousands of times a day — every text, image, and video is compressed before it travels between devices (GZIP, JPEG, PNG, H264…). These schemes look domain-specific, but they share deep, common ideas. Our goal: build intuition for those fundamentals.

Let's start with a deliberately simple scenario.

We have a large text file whose every character is drawn independently and uniformly from S = {A, B, C, D}, i.e. P(A) = P(B) = P(C) = P(D) = ¼. Say the file has 1,000,000 letters.

Stored as ASCII, each symbol takes 8 bits, so the file is 1,000,000 bytes. But the four ASCII codes all begin with the same 1000… — redundant when we only have four symbols. The natural question: can we make the file smaller, losslessly? We restrict ourselves to symbol codes — a fixed binary codeword per symbol.

Fixed-bitwidth encoding

With only k distinct symbols we need just ⌈log₂ k⌉ bits each. Drop unused symbols and the cost shrinks. Toggle which symbols actually appear:

So a uniform 4-symbol file compresses 4× for free. But this only helps when symbols are completely absent. What if every symbol appears, just in very different proportions?

Quiz 2 — Beyond fixed bitwidth With P(A) = P(B) = 0.49 and P(C) = P(D) = 0.01, can a symbol code beat 2 bits/symbol?

Fixed-width can't: it always spends ⌈log₂ 4⌉ = 2 bits regardless of probabilities. But intuitively we mostly see A and B, so we'd like to approach 1 bit/symbol. The fix: variable-length codes — short codewords for frequent symbols, longer for rare ones.

With the 0.49 / 0.49 / 0.01 / 0.01 split this gives E[L] = 1.53 bits/symbol — worse than 2 in the worst case (C, D cost 3), but better on average. Exactly the trade we wanted.

Why is this even decodable?

Fixed-width decoding was trivial: chop the stream every 2 bits. With variable lengths we don't know where one codeword ends and the next begins. Let's encode a string and then decode it bit-by-bit to see what makes it work.

The decoder reads bits until the accumulated string matches a codeword, emits that symbol, and resets. It works because no codeword is a prefix of another — the moment we match, we can commit with zero ambiguity, without peeking at future bits.

Key idea — Prefix-free codes. If no codeword is a prefix of any other, the symbol code is prefix-free (a.k.a. instantaneous). Prefix-free codes are always losslessly and instantaneously decodable.

Prefix-freeness is a sufficient condition, not the only way to be lossless — but it gives the cleanest decoder. Test codes below; the second preset is lossless yet not prefix-free, which breaks the simple decoder.
