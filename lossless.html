How Lossless Compression Works

Interactive

Make a file smaller — without losing a single bit.

Text, music, and video are all just sequences of symbols stored as bits. Lossless compression is the art of writing the same information with fewer bits, so you can perfectly rebuild the original. Here's the whole idea — and a lab to try it on your own text.

Step 1

Everything is bits

A computer can only store 0s and 1s. To store letters, we agree on a code: a mapping from each symbol to a pattern of bits. Standard text uses ASCII, where every character is 8 bits:

A→ 01000001   B→ 01000010

But 8 bits buys you 256 different patterns. If your file only uses a handful of distinct symbols, that's wildly wasteful. With an alphabet of $k$ symbols, you only need enough bits to give each one a unique pattern:

$\text{bits per symbol} = \lceil \log_2 k \rceil.$

Four symbols? $\lceil\log_2 4\rceil = 2$ bits each. Just A and B? One bit each. This is fixed-length coding, and it's the baseline to beat.

Step 2

The trick: not every symbol is equally common

In English, e and the space character appear constantly, while z and q are rare. So why spend the same number of bits on each? Give common symbols short codes and rare symbols longer ones, and the average shrinks.

Morse code figured this out in 1838: the most common English letter, E, is a single dot ·, while Q is — — · —. Same idea, 180 years earlier.

To do this well, we need to know how often each symbol shows up — its probability distribution. Frequencies are the raw material of compression: everything downstream is built from them.

Step 3

Entropy: the speed limit

Here's the beautiful part. Once you know the distribution, there's a hard limit on how few bits per symbol any code can use, on average. It's called the entropy:

$\displaystyle H = \sum_i p_i \,\log_2\frac{1}{p_i} \quad\text{bits per symbol.}$

Read $\log_2\frac{1}{p_i}$ as the "surprise" of seeing symbol $i$: rare symbols are surprising and cost more bits; common ones are predictable and cost less. Entropy is just the average surprise — and it's the best any compressor can do.

The fine print: this limit assumes each symbol is independent of the last (the "i.i.d." assumption). Real text isn't — q is almost always followed by u — so smart compressors that model patterns can beat single-symbol entropy. But it's the perfect north star for our first compressor.

What's next

Huffman gives the best possible code when each symbol gets a whole number of bits. But you may have noticed it still can't quite reach the entropy floor — because a symbol that "deserves" 1.4 bits has to round up to 2. Beating that requires letting symbols share fractional bits, which is exactly what arithmetic coding and ANS do. More demos coming.
